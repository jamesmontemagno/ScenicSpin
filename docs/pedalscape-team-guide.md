# ScenicSpin team guide

Last updated: 2026-06-22

## Product positioning

- ScenicSpin is the shared source repository for two sister, local-first scenic fitness PWAs:
  - PedalScape: indoor cycling, live at `https://pedalscape.com/`.
  - BeltScape: treadmill walking, live at `https://beltscape.com/`.
- Public promise:
  - PedalScape: "Any bike. Any screen. Beautiful rides from home."
  - BeltScape: "Any treadmill. Any screen. Beautiful walks from home."
- Both products intentionally avoid accounts, analytics, backend databases, and social login.
- The core job is simple: browse curated scenic YouTube routes, pick one, and launch a fullscreen player on a laptop, tablet, or TV.

## Live sites, repo, and hosting

- GitHub repo: `https://github.com/shanselman/ScenicSpin`.
- Each site is built from shared templates plus a site config in `sites/`.
- Deployment builds each site into `dist/<site>/` and publishes it to that site's GitHub Pages repo.
- Custom domains are generated by the build from site config:
  - PedalScape: `pedalscape.com`
  - BeltScape: `beltscape.com`
- The repo/package still contain some legacy `ScenicSpin` / `scenic-ride-catalog` naming; user-facing branding is site-specific.

## Architecture

- Shared static templates: `index.html`, `src/styles.css`, `src/app.js`, `manifest.webmanifest`, `service-worker.js`, locale JSON, icons, assets, and route data.
- Site configs:
  - `sites/pedalscape.config.json`
  - `sites/beltscape.config.json`
- Production catalogs:
  - PedalScape source: `routes/catalog.json` (currently 40 rides)
  - BeltScape source: `routes/beltscape-catalog.json` (currently 24 walks)
  - Build output always uses `routes/catalog.json` inside each `dist/<site>/` so the app stays site-agnostic.
- Candidate/review backlog: `routes/candidate-backlog.json`.
  - It is shared review data, not a production catalog.
  - Current active candidate count: 0.
  - Previously promoted PedalScape candidates were cleared from `candidateRoutes` on 2026-06-22 and summarized in `promotedArchiveSummary` so hidden review mode does not show already-promoted work as active backlog.
- Hidden review mode:
  - Open with `?review=1` or `#review`.
  - Loads only active candidates from `candidateRoutes` for triage.
  - Lets reviewers mark Promote/Yes, Reject/No, or Defer/Maybe, add notes, and export decisions when candidates exist.
- PWA:
  - The manifest and service worker are tokenized per site.
  - The app shell, catalog JSON, candidate backlog JSON, icons, and core assets are cached.

## Data model and curation workflow

- Catalog route fields include:
  - `id`, `title`, `sourceUrl`, `embedUrl`, `sourcePlatform`, `creator`, `location`
  - `durationMinutes`, `difficulty`, `terrain`, `sceneryTags`
  - `videoQuality`, `audio`, `cameraStyle`, `embeddingAllowed`, `license`, `curationNotes`
- Candidate entries add review/curation fields such as `status`, `curationTier`, `promotionReadiness`, `verification`, `productionCatalogId`, and `promotedToCatalogAt`.
- Keep `candidateRoutes` for active review only. When every candidate in a batch has already been promoted or rejected, remove the stale review cards and keep only a concise archive/decision summary if it helps future maintainers.
- Promotion heuristics:
  - Prefer official creator/public channel sources, not reposts.
  - Confirm embeds work without downloading or rehosting.
  - Verify location, duration, audio, camera stability, quality, and workout pacing.
  - Promote for catalog diversity across geography, duration, difficulty, terrain, season, and audio style.
  - Positive triage signals include about-30-minute duration, no interruptions, high-quality video, and substantial view count, but these do not replace legitimacy, embed, and legal checks.
- Badge/chicklet philosophy:
  - Use compact, useful badges over raw metadata dumps.
  - Keep route cards visually scannable.
  - Strip reviewer-only text like "verify", "before launch", "do not download", and similar internal notes from public card copy.
  - Limit media badges to five.
- Legal/platform stance:
  - Link/embed official public streams only.
  - Do not download, proxy, mirror, or rehost creator video files.
  - Re-check embed availability, source authenticity, licensing, and platform terms before launch because platform settings can change.

## UX principles learned

- Visual-first cards: thumbnails/preview art lead each route card, with title, location, duration, intensity, and a few useful badges.
- Compact hero: the hero highlights a recommended first route or a "Continue" state without overwhelming browsing.
- Sticky search/filter: filters stay close to the top of the experience and persist locally.
- Hidden review mode: backlog triage exists for maintainers but is not mixed into public catalogs.
- Local-first trust footer: explain that there is no account or backend, and that favorites/preferences stay in this browser.
- Useful curation badges: surface high-signal badges like `4K`, `60+ min`, `Climb`, `Mountains`, `Water/Lakes`, `Natural audio`, or `No music`; avoid noisy/internal review labels on production cards.

## Local-first and PWA behavior

- Favorites, recent routes, selected/continue route, and filters are stored locally in the browser.
- Local data controls:
  - Save/remove favorites.
  - Track up to five recent routes.
  - Continue the last started route from the hero.
  - Export/download local backup JSON.
  - Copy backup JSON.
  - Import a validated ScenicSpin backup for the current site.
  - Reset local data.
- Local storage/session keys are namespaced by site where appropriate:
  - `scenicRideCatalog.selectedRouteId` in `sessionStorage` and `localStorage`
  - `scenicRideCatalog.favoriteRouteIds`
  - `scenicRideCatalog.recentRouteIds`
  - `scenicRideCatalog.filterPreferences`
  - `<SiteName>.reviewDecisions`, for example `PedalScape.reviewDecisions`
- Backup format:
  - `app` is the site name.
  - `schemaVersion: 1`
  - `localData.selectedRouteId`
  - `localData.favoriteRouteIds`
  - `localData.recentRouteIds`
  - `localData.filterPreferences`
- Offline/install behavior:
  - The app shell, catalog JSON, backlog JSON, and icons are cached by the service worker.
  - The app can render the shell/catalog offline after caching.
  - Ride/walk videos and YouTube thumbnails are third-party network resources and still require internet.

## Testing and validation

- Install dependencies: `npm install`
- Lightweight JavaScript syntax check: `npm run check`
- Build both sites: `npm run build:all`
- End-to-end tests:
  - `npm run test:pedalscape`
  - `npm run test:beltscape`
  - `npm run test:all`
- Local built-site previews:
  - `npm run preview:pedalscape` serves `http://127.0.0.1:5173`.
  - `npm run preview:beltscape` serves `http://127.0.0.1:5174`.
- Route/manifest/config JSON parse sanity check:
  - `node -e "const fs=require('fs'); for (const file of ['routes/catalog.json','routes/beltscape-catalog.json','routes/candidate-backlog.json','manifest.webmanifest','package.json','sites/pedalscape.config.json','sites/beltscape.config.json']) JSON.parse(fs.readFileSync(file,'utf8'))"`
- Current e2e coverage includes:
  - Loading each site's JSON catalog and PWA assets.
  - Clean production badges without review metadata.
  - Scenery/terrain badge normalization.
  - Hidden candidate review mode, including empty backlog behavior and review decision export when candidates exist.
  - Favorites persistence and favorites-only filter.
  - Search/filter persistence.
  - Continue route, recents, hero state, and iframe loading.
  - Reset local data.
  - Install prompt handling.
  - Export/import validation.
  - Service-worker offline app shell and route data.

## Social and branding

- Per-site metadata and icons are driven by `sites/*.config.json` and the matching `icons/<site>/` directory.
- Shared/source icon assets may still exist at the root of `icons/`, but production builds should use site-specific generated assets.
- OG image source assets live in `assets/`.
- OG image generation script: `scripts/generate-og-image.ps1`.

## Known caveats and next queue

- Custom playlists are a later feature; today the app is catalog + selection + continue/favorites/recents.
- There is no formal JSON Schema validation yet; validation is currently syntax checks plus e2e assumptions.
- Keep the candidate backlog focused on active review items; do not let promoted catalog entries accumulate as confusing review-mode noise.
- Continue curating both catalogs, but keep public catalogs curated rather than raw/backlog-sized.
- Consider splitting `src/app.js` into modules before the app grows much further or before the catalog/review workflow expands past 100+ reviewed routes.
- YouTube embeds and thumbnails are third-party resources; availability, embed settings, thumbnails, ads, and playback behavior can change outside this repo.
- Keep product naming aligned over time: repo/monorepo name is ScenicSpin, user-facing brands are PedalScape and BeltScape.
